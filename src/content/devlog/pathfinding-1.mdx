## The Game

During development of Pathways & Poltergeists we built a number of mechanics that kept breaking our pathfinding algorithm. The core of the game requires the player to build a connecting path using the provided Tetris-like pieces. Everytime a player block is dropped, we rerun the breadth first search (BFS) until a connection is found. While standard BFS worked perfectly for our simple grid based puzzle game, its limitations became very apparent as more complex mechanics were added.
 
Throughout this series we'll discuss each of the modifications made to have a hybrid BFS algorithm capable and ranking objectives, simulating failure states, and coordinating with other agents.

---

## OG BFS

BFS is pretty straightforward. It searches by expanding out from the origin position in all directions, checking every tile in a layer before moving to the next layer. It works similar to how you'd scan around a large public space looking for the exit.

<SonarAnimation />

In the animation above, you can imagine the blue tiles as the path the algorithm is checking. The orange tile is the exit. Your eyes keep scanning around farther and farther until it finds the green exit sign.

```
// Standard BFS

function findPath(start, exit, grid)
    queue   = [start]
    visited = {start}
    parents = empty map

    while queue is not empty
        current = queue.dequeue()

        if current is exit
            return reconstructPath(parents, start, exit)

        for each neighbor of current
            if neighbor is walkable and neighbor not in visited
                add neighbor to visited
                parents[neighbor] = current
                queue.enqueue(neighbor)

    return NO_PATH_FOUND
```

Again this initially worked fine for Pathway & Poltergeists. However, that all changed as we began to add waypoints, teleporters, and switches.

## Waypoints and Switch/Gate Mechanics

### The Problem

Suddenly now the player needs to connect a path from the ghost to the waypoint before reaching the exit. With standard BFS it would simply ignore the waypoint and go straight to the exit.

Imagine if you went inside a mall and just went straight for the exit, then you'd never reach the store you went there for in the first place.

<SnackStoreAnimation />

### The Solution

We fixed this by simply adding each waypoint into a list of "must visit" or "required" blocks prior to exit. The algorithm then scans out and initially looks for waypoints. As it reaches a waypoint, it just checks it off the list, and runs again from that waypoint until all waypoints on the list are found. Then it simply looks for the exit.

```
// State-Aware BFS — each state now tracks position AND progress
// State = { position, collectedWaypoints }

while queue is not empty
    current = queue.dequeue()

    // Only finish if we're at the exit AND have hit every waypoint
    if current.position is exit and current.collectedWaypoints contains all waypoints
        return reconstructPath(parents, startState, current)

    for each neighbor of current
        newProgress = current.collectedWaypoints
        if neighbor is a waypoint
            add neighbor to newProgress

        // "Cell A with 0 waypoints" is a different state than "Cell A with 1 waypoint"
        nextState = { position: neighbor, collectedWaypoints: newProgress }

        if neighbor is walkable and nextState not in visited
            add nextState to visited
            parents[nextState] = current
            queue.enqueue(nextState)
```

Each level now includes a full list of all waypoints. We run the BFS search until the list of required waypoints is satisfied and we reach the exit.

Teleporters are not required, but change where the BFS "path" is located. Using a teleporter means moving to a new location, so the BFS has to account for this in the pathway by updating where it searches from. Under the hood the client really doesn't know what a teleporter is though, it only knows that it's moved a considerable distance (greater than 1 unit of space).

Next we add switches and locked gates. Naturally we want the pathfinder to prioritize hitting switches. In fact we should prioritize them over waypoints right? This way any waypoints that were behind locked doors will now be accessible.

```
// Switch/Gate Support — state now also carries which switches are active
// State = { position, collectedWaypoints, activeSwitches }

for each neighbor of current
    newSwitches = current.activeSwitches

    if neighbor is a switch
        add neighbor to newSwitches         // flip it on

    if neighbor is a gate
        if matchingSwitch for gate not in newSwitches
            skip this neighbor              // gate is locked, path is blocked

    // Layer onto the state from before — nothing else changes
    nextState = {
        position:            neighbor,
        collectedWaypoints:  current.collectedWaypoints,
        activeSwitches:      newSwitches
    }

    if neighbor is walkable and nextState not in visited
        add nextState to visited
        parents[nextState] = current
        queue.enqueue(nextState)
```
